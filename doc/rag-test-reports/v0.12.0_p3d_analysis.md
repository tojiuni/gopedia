## 시도한 FTS 접근법

### 시도 1: plainto_tsquery AND 로직
초기 구현. `plainto_tsquery('simple', query)` → 모든 토큰 AND 조건.  
"ost-stor-01 storage hostname" → `'ost-stor' & 'ost' & 'stor' & '01' & 'storage' & 'hostname'`  
→ 모든 토큰 동시 존재 요구 → MISS 청크 0건 반환 → **완전 실패**

### 시도 2: OR 로직 전환 (전체 토큰)
`to_tsquery` + OR 연결자로 변경.  
"ost-stor-01 storage hostname" → `'ost-stor-01' | 'ost' | 'stor' | '01' | 'storage' | 'hostname'`  
→ 162건 매칭. FTS limit=30에서 MISS 4/6 회수 가능.  
→ 그러나 한국어 토큰("있는가", "설치한다고")도 OR에 포함 → 과도한 매칭 → **P@3 0.272 (회귀)**

### 시도 3: 기술 토큰 필터 (ASCII 우세 토큰만)
한국어 문자 비율 > 40%인 토큰 제외, 한국어 particle/suffix 제거.  
"Kubernetes는" → "kubernetes" (는 제거)  
"역할(Cinder" → 분리 처리 → "cinder"만 유지  
→ FTS 매칭 수 감소. MISS 4/6 여전히 회수 가능(FTS limit=60).  
→ 그러나 "openstack", "ubuntu", "kubernetes" 같은 영어 기술 용어들이 corpus 전체에 분포  
→ 여전히 100-200건 매칭 → **P@3 0.278 (회귀)**

## 실패 원인 분석

### 핵심 문제: corpus 전반 기술 용어의 OR 매칭

```
질의: "Osteon 00 Overview ... Ubuntu 24.04 LTS (Base OS), OpenStack Dalmatian"
추출 기술 토큰: 'osteon' | '00' | 'overview' | 'ubuntu' | '04' | 'lts' | 'base' | 'openstack' | 'dalmatian'
FTS 매칭 건수: 188개 (OpenStack 가이드 전체에 분산)
```

이 188개가 RRF로 dense 결과와 동등하게 경쟁 → primary qrel(dense rank 1)이 top-5 이탈.

### RRF alpha=0.5 균등 가중치 문제

```
dense rank 1 (primary qrel):  alpha/(k + 1) = 0.5/61 ≈ 0.0082
FTS rank 1 (irrelevant doc): (1-alpha)/(k + 1) = 0.5/61 ≈ 0.0082
```

FTS rank 1이 dense rank 1과 동점 → FTS 상위 4개 오류 문서가 primary qrel을 rank 5+ 밖으로 밀어냄.

### alpha 딜레마

| alpha | FTS 유일 MISS 회수 가능? | Primary qrel 회귀 발생? |
|-------|------------------------|----------------------|
| 0.5 (균등) | 가능 (FTS 가중치 충분) | 발생 (FTS 오류 문서 경쟁) |
| 0.85 | 불가능 (FTS 점수 < dense 최하위) | 없음 |
| 0.9+ | 불가능 | 없음 |

어떤 alpha를 사용해도 precision과 recall을 동시에 개선할 수 없음.

## 결론 및 다음 단계

### 현황 (v0.12.0)
- 하이브리드 비활성 상태로 복귀 (`GOPEDIA_HYBRID_SEARCH_ENABLED=false`)
- hybrid 인프라 코드 (`pg_fts_search_l3`, `rrf_fuse`, `_build_or_tsquery`) 코드베이스 유지
- 향후 Qdrant sparse vector 연동 시 기반 활용 가능

### 정식 P3-D를 위한 요건
1. **Qdrant sparse vector 인덱싱**: BM25 계산이 ingest 시점에 수행 → 검색 시 정확한 term frequency 반영
2. **re-ingest 필요**: 전체 문서를 sparse vector와 함께 재색인
3. **sparse-dense 융합**: Qdrant 자체 hybrid 검색 API 또는 RRF

### 대안 검토 순서
1. **IMP-17 (청킹 개선)**: heading+fixed-size 하이브리드 → MISS 청크 경계 문제 해결 가능
2. **P3-D 정식 (Qdrant BM25 + re-ingest)**: 구조적 keyword 매칭 문제 해결
