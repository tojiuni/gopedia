## 0. Re-Ingest 목적

이전 v0.6.0 eval은 변경 전 chunking으로 만들어진 인덱스를 사용했기 때문에
ordered list 그룹화 및 ByFixedSizeChunker 1000자 변경 효과를 측정할 수 없었다.
이번 eval은 **새 chunker로 universitas 전체를 재인제스트**한 후 측정한 결과다.

### 재인제스트 절차

1. gopedia API 종료 (pid 1913934)
2. `go build -o /tmp/gopedia-api ./cmd/gopedia/` — 최신 바이너리 빌드 (40.6 MB)
3. Postgres 전체 테이블 TRUNCATE, Qdrant 컬렉션 삭제
4. 새 gopedia API 시작 → Qdrant 컬렉션 자동 재생성 (1024-dim, Cosine)
5. `ingest_universitas.sh` 실행 (67 files, 67/67 OK, DELAY=0.3s)
6. gardener qrel 재해결: `POST /datasets/{id}/resolve-qrels` → 44/44 resolved
7. Eval 실행

## 3. 근본 원인 분석

### 3-1. Gopedia 아키텍처 회귀 (-4 hits)

`q_gopedia_clickhouse_role`, `q_gopedia_l1_role`, `q_gopedia_postgresql_role`, `q_gopedia_typedb_role` — 4개 모두 miss.

**가능한 원인 두 가지**:
1. **qrel 재해결 오류**: 재인제스트 후 `resolve-qrels`가 유사한 내용의 청크 중 다른 것을 선택했을 가능성. 예를 들어 `q_gopedia_typedb_role`과 `q_gopedia_l1_role`은 동일한 qrel target(`8c182ed8`)으로 해결됨 — 두 쿼리가 다른 청크를 대상으로 해야 하는데 같은 청크로 해결된 것.
2. **Chunking 변화**: 1000자 ByFixedSizeChunker로 chunking이 거칠어져 역할별 세분화가 줄어든 것.

### 3-2. `q_registry_push_steps` — ordered list 그룹화 미적용

`registry-push.md`의 ordered list (Steps 1~4: Login → Build → Push → Verify)가 L3 인덱스에 없음:

```sql
-- "docker login" 포함 L3 청크 없음 (0 rows)
SELECT id, LEFT(content, 200) FROM knowledge_l3 WHERE content ILIKE '%docker login%';
```

**원인 추정**: 1000자 ByFixedSizeChunker가 code block을 포함한 ordered list 항목을 별도 청크로 나누지 않고 합쳐서, Qdrant에 임베딩된 청크의 semantic center가 달라졌을 수 있음. 또는 contextual embedding 과정에서 해당 청크가 필터링되었을 가능성.

### 3-3. qrel 재해결(Re-Resolution) 품질 문제

`q_registry_push_steps` qrel target이 `67d0811e`으로 해결됨:
```
content: "- Registry: https://registry.toji.homes/\n- Registry UI: https://registry-ui.toji.homes/"
```
실제 원하는 내용(push steps)과 전혀 다른 청크. Resolver가 "registry.toji.homes" 키워드 매칭으로 잘못된 청크를 선택한 것.

이는 새 chunking + qrel 재해결의 복합 오류다:
- 실제 push steps 청크가 없거나 약한 임베딩 → Resolver가 대신 URL 청크를 선택

### 3-4. 인덱스 규모 축소 효과

| 버전 | L3 chunks | Qdrant vectors |
|------|-----------|----------------|
| v0.5.x (이전) | 3,983 | 3,273 |
| v0.6.0 re-ingest | 820 | 666 |

청크 수 약 4.8배 감소. 세밀한 내용 검색에서 precision이 떨어질 수 있음. 이 규모 차이의 원인은 추가 분석 필요.
