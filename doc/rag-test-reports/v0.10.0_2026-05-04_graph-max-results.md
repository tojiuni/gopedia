## 분석: GRAPH_MAX_RESULTS=3 효과

### 핵심 결론

`GRAPH_MAX_RESULTS=3` / `GRAPH_MAX_SIBLINGS=3` 설정이 **모든 IR 지표에 영향 없음**.  
Recall@5=0.5인 쿼리 수도 v0.9.0과 동일(6개).

### Recall@5=0.5 쿼리 현황

| 버전 | Recall@5=0.5 쿼리 수 |
|------|----------------------|
| v0.8.1 (TypeDB off) | 7 |
| v0.9.0 (TypeDB on, 무제한) | 6 |
| v0.10.0 (TypeDB on, MAX=3) | 6 |

→ graph expansion 결과 수를 줄여도 동일 6개 쿼리가 secondary qrel 미검색.

### P@3 불변 분석

P@3 = 0.367 유지. **graph 결과 "수"가 아니라 "삽입 위치"가 P@3 하락의 원인**:

- cross-encoder 재랭킹 후 graph expansion 결과가 rank 1-3 내에 들어가면서 secondary qrel 탈락
- 결과 수를 3개로 제한해도 재랭킹 순위 경쟁 자체는 동일하게 발생
- P@3 구조적 상한: `(30 + k) / 90`, 현재 k=3 → 0.367 (gardener top-3 실측 API 없이 k 개선 불가)

### ragas/context_relevance 소폭 개선 (+0.008)

graph expansion 결과 수를 제한하자 컨텍스트 관련성 평균이 0.900 → 0.908로 소폭 향상.  
삽입되는 sibling 문서 품질이 더 선별됨에 따른 효과로 추정.

## 결론

`GRAPH_MAX_RESULTS=3` 설정은 **안전한 운영 기본값**으로 유효:

- **긍정**: ragas/context_relevance +0.008. Recall@5·MRR·nDCG 유지.
- **중립**: P@3 불변. 상한 설정으로 P@3 회복 불가 — 구조적 한계 재확인.
- **부정**: 없음. 기존 Recall@5 개선(+0.017 vs v0.8.1) 유지.

운영 환경에 `GRAPH_MAX_RESULTS=3` / `GRAPH_MAX_SIBLINGS=3` 고정 권장.
