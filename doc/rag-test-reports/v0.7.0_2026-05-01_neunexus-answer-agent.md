## 0. 이번 버전 변경 내용

### 신규 기능: 계층형 RAG Answer Agent (`/api/answer`)

LLM(gemma4:26b)이 tool calling으로 지식 계층(L3 → L2 → L1)을 직접 탐색하여 합성 답변을 반환하는 에이전트.

**아키텍처:**
```
GET /api/answer?q=<query>
  └─ Go: python 서브프로세스 spawn
       └─ flows.xylem_flow.cli answer --query <q>
            └─ answer_agent.run(query, conn)
                 └─ [LLM tool-calling loop, max 8회]
                      ├─ search(query, top_k) → retrieve_and_enrich()
                      ├─ restore_l2(l2_id) → restore_code_for_l2()
                      ├─ restore_l1(l1_id) → restore_content_for_l1()
                      ├─ answer(content, sources) → 종료
                      └─ not_found(reason) → 종료
```

**관련 파일:**
- `flows/xylem_flow/answer_agent.py` (신규)
- `internal/api/api.go` — `GET /api/answer` 엔드포인트 추가
- `internal/api/types.go` — `AnswerResponse` 구조체 추가
- `deploy/k8s/gopedia-svc.yaml` — `OLLAMA_CHAT_URL`, `OLLAMA_CHAT_MODEL` env 추가

**gopedia-bot 연동 (`neunexus` PR #75):**
- `rag.query` capability가 `/api/answer` 우선 호출 → 실패 시 `/api/search` fallback

## 2. 성능 집계

### 2-1. iterations / 토큰 효율성

| 질문 | iterations | 결과 | 추정 입력 토큰 |
|------|-----------|------|--------------|
| Q0 GeneSo | 5 | ✅ | ~11,000 |
| Q1 K8s | 6 | ✅ | ~13,000 |
| Q2 Network Design | 8 (초과) | ❌ | ~16,000 (낭비) |
| Q3 Troubleshooting | 5 | ✅ | ~10,000 |
| Q4 Brand Guidelines | **3** | ✅ | **~3,500** |
| Q5 AWS (없음) | **2** | ❌ (정상) | **~800** |

> **이상적인 케이스 (l3만으로 충분):** ~400–1,000 토큰  
> **현재 평균 (성공 케이스):** ~9,375 토큰  
> **개선 목표:** P1 개선 항목 적용 시 ~2,000–3,000 토큰으로 감소 예상

### 2-2. 탐색 패턴 분류

| 패턴 | 발생 | 원인 |
|------|------|------|
| search → answer (이상적) | 0/5 | snippet 부족으로 전혀 발생 안 함 |
| search → restore_l2 → answer | 1/5 (Q4) | 비교적 단순한 factual 질문 |
| search → restore_l2 × n → restore_l1 → answer | 3/5 | 복합 질문 |
| search 루프 → 초과 | 1/5 (Q2) | search 쿼리 재시도 반복 |

### 2-3. 근본 원인

1. **snippet 300자 제한** — LLM이 평가할 컨텍스트가 부족 → 즉각적인 escalation
2. **l1_id 기준 dedup** — 같은 문서의 여러 섹션이 1개로 축소 → 정보 부족
3. **l2_summary 미전달** — `retrieve_and_enrich()`에서 이미 fetch한 섹션 요약을 answer_agent가 버림
4. **search 쿼리 재시도 루프** — 스니펫 불충분 시 다른 쿼리로 재검색 → iteration 낭비

## 3. 개선 방향 (우선순위)

### P1 — 코드 수정만으로 즉시 적용 (재인덱싱 불필요)

→ [IMP-09], [IMP-10], [IMP-11] 참조 (`doc/IMPROVEMENTS.md`)

- **IMP-09**: search 결과에 `l2_summary` + `surrounding_context` 포함 (snippet 300 → 500자)
- **IMP-10**: dedup 기준을 `l1_id` → `l2_id`로 변경
- **IMP-11**: 시스템 프롬프트 튜닝 — "충분하면 즉시 answer" 강조

### P2 — 아키텍처 개선

→ [IMP-12], [IMP-13] 참조

- **IMP-12**: Python 상주 gRPC 서비스 전환 (subprocess → persistent service)
- **IMP-13**: Query Rewriting — 한국어 구어체 → 임베딩 친화적 기술 용어 변환

### P3 — 재인덱싱 필요

→ [IMP-14], [IMP-15] 참조

- **IMP-14**: L2 summary Qdrant 인덱싱 (L3 + L2 hybrid 검색)
- **IMP-15**: Cross-Encoder Reranker 활성화 (`--reranker` 플래그, 이미 구현됨)

## 4. 결론

- **Answer Agent 기본 동작 확인**: 계층형 탐색(L3→L2→L1) 및 LLM 합성 답변이 정상 동작.
- **없는 정보 처리**: not_found 케이스를 빠르게(2 iterations) 정확히 판별.
- **주요 병목**: 스니펫 크기 부족으로 인한 과도한 restore escalation — search 결과만으로 답변 가능한 케이스가 0%.
- **다음 액션**: P1 IMP-09~11 즉시 적용 → 같은 쿼리로 재측정하여 iterations/토큰 감소 확인.
