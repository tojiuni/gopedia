## Cross-Encoder Reranker 효과 분석

### P@3 불변 — 핵심 결론

**P@3 = 0.367로 v0.10.0과 동일.** reranker 활성화로 P@3 개선 없음.

**원인 분석**: reranker는 top-k 후보 내에서 순위만 재조정. secondary qrel이 Qdrant 벡터 검색의 top-k 후보에 아예 포함되지 않는다면 reranker로 구제 불가.

```
P@3 = 0.367 = 33/90
  - secondary 15개 중 top-3 내 검색: 3개 (k=3, 변화 없음)
  - Recall@5 < 1.0 쿼리: 6개 (v0.10.0 동일)
```

→ **secondary qrel이 top-k 후보 자체에 없음**으로 최종 확인.  
→ 벡터 검색(BGE-M3) 단계에서 이미 secondary가 탈락하는 구조.

### 레이턴시 개선 (+512ms 단축)

reranker 활성화 후 오히려 평균 레이턴시 감소: ~1,272ms → ~760ms.

```
원인: reranker 활성화 시 pipeline 변경
  reranker 비활성: Qdrant 30개 → fetch_rich_context(30개) → return 5개
  reranker 활성:   Qdrant 30개 → rerank → fetch_rich_context(5개) → return 5개

  fetch_rich_context(PostgreSQL): 30개 vs 5개 → 쿼리 수 6분의 1
  reranker 추가 오버헤드 < PostgreSQL fetch 절감
```

→ **reranker 활성화가 레이턴시 측면에서도 유리**. 유지 권장.

### MRR@10·Recall@5 불변

Primary qrel은 reranking 후에도 rank-1 유지 (MRR@10=0.950).  
Recall@5도 동일 — top-k 후보 구성이 변하지 않으므로 recall은 영향 없음.

## 결론

| 항목 | 결과 |
|------|------|
| P@3 개선 효과 | ❌ 없음 — secondary qrel이 top-k 후보 자체에 부재 |
| 레이턴시 | ✅ -512ms 개선 — fetch_rich_context 절감 효과 |
| 안정성 | ✅ 모든 IR 지표 유지 |

**reranker는 P@3를 개선하지 않지만 레이턴시 개선 효과가 있으므로 유지.**  
**P@3 개선 다음 단계**: P3-A(gardener top-k API) 데이터로 secondary qrel의 실제 rank 확인 후 접근법 결정.
