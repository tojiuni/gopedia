## 1. 브랜치 주요 변경사항

### `feat/reranker-adaptive-chunking`

#### Cross-Encoder Reranking (`flows/xylem_flow/retriever.py`)
- Qdrant dense 검색 (top-30 candidates) → Cross-Encoder 재정렬 → final top-5 cutoff
- 모델: `BAAI/bge-reranker-v2-m3` (multilingual, Korean 지원)
- `use_reranker=True` 파라미터 또는 `GOPEDIA_RERANKER_MODEL` 환경변수로 활성화
- CLI: `--reranker` / `--reranker-model` 플래그 추가
- API: `reranker=true` / `reranker_model` 쿼리 파라미터 추가

#### 적응형 청킹 — Ordered List 그룹화 (`internal/phloem/chunker/structured.go`)
- **변경 전**: 각 ordered list 항목이 개별 L2 청크 (e.g., push 절차 5단계 → 5개 청크)
- **변경 후**: 연속 ordered list 항목을 하나의 L2 청크로 그룹화
- 절차형 문서(push 순서, 설치 단계)의 단계가 같은 L2에 속해 neighbor window로 전체 맥락 복원 가능

#### ByFixedSizeChunker 기본 크기 확대 (`internal/phloem/chunker/fixed.go`)
- `MaxChars` 기본값: 500 → 1000 chars

## 4. 근본 원인 분석 — 왜 Reranker가 지표를 개선하지 못했는가

### Dense Retrieval Bottleneck

Cross-Encoder Reranking은 **Qdrant top-30 후보 내에서 순위를 재정렬**하는 단계다.
미스된 17개 쿼리를 분석하면 공통 패턴이 나타난다:

- Qdrant top-5의 `is_relevant=false` 비율: 17/17 쿼리 전체 (5/5 irrelevant)
- Qdrant top-1 score: 0.84~0.90 (높은 점수이지만 관련 없는 문서)
- 결론: **관련 청크가 Qdrant top-30에 포함되지 않아** reranking이 효과 없음

Reranker는 이미 dense search가 찾아온 후보를 재정렬하는 역할이다.
후보 pool에 정답이 없으면 reranker로 순위를 아무리 바꿔도 recall이 올라가지 않는다.

### Ordered List 그룹화 — Re-ingest 필요

`structured.go`의 ordered list 그룹화는 **새로 인제스트하는 문서에만 적용**된다.
현재 Qdrant 인덱스는 변경 전 chunking으로 만들어진 벡터를 그대로 사용 중이므로,
이번 eval에서는 chunking 변경 효과를 측정할 수 없었다.

## 7. 권고사항 및 다음 단계

### 즉시 적용 가능

| 우선순위 | 권고 | 예상 효과 |
|---------|------|----------|
| **HIGH** | 변경된 chunking으로 universitas 문서 **전체 re-ingest** | ordered list 그룹화 효과 측정 |
| **HIGH** | `candidate_limit` 30 → 60~100으로 상향하여 reranker 효과 극대화 | Dense bottleneck 완화 + reranker ROI 증가 |
| **MED** | BM25 hybrid search (Qdrant sparse vector + RRF) | 숫자·버전·키워드 쿼리 miss 감소 |

### Dense Retrieval Gap 해소

현재 17개 miss는 모두 dense vector similarity가 충분히 높지 않은 경우다.
Reranker가 도움이 되려면 **먼저 candidate pool에 정답이 있어야 한다**.
`candidate_limit` 상향 → Reranker가 실질적으로 순위를 교정할 기회 증가.

```
현재: Dense top-30 → Reranker → top-5
권고: Dense top-60 → Reranker → top-5 (정답 포함 확률 ↑)
```

### Reranker Latency 최적화

현재 reranker는 검색 요청마다 Python subprocess에서 모델을 재로드한다.
레이턴시 최적화를 위해 **별도 reranker gRPC 서비스**로 분리하는 것을 권고한다:

```
Gopedia API → reranker service (persistent process, model loaded once)
                              → score 30 pairs → return ranked IDs
```
