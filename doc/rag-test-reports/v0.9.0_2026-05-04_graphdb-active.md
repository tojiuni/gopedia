## TypeDB Graph Context 효과 분석

### Recall@5 개선 (+0.017)

| 버전 | Recall@5=0.5 쿼리 수 | 해당 쿼리 |
|------|----------------------|-----------|
| v0.8.1 (TypeDB off) | 7 | osteon_01, 05, 07, 13, 17, 21, 27 |
| v0.9.0 (TypeDB on) | 6 | osteon_05, 07, 13, 21, 24, 27 |

- **개선**: osteon_01, osteon_17 2개 모두 graph_context 확장으로 secondary qrel이 top-5 진입
- **퇴보**: osteon_24가 v0.8.1에서 top-5 성공이었으나 v0.9.0에서 0.5로 변경 (graph 결과 삽입으로 secondary 밀려남)
- **순 개선**: +2개 개선 -1개 퇴보 = 순 +1개 → Recall@5 +0.017

### P@3 소폭 하락 (-0.022)

Graph expansion 결과(`source: "graph_expansion"`)가 결과 목록에 삽입되면서 일부 secondary qrel이 top-3 밖으로 밀려남.

```
P@3 = 0.367 = 33/90
  - 3개 쿼리: secondary top-3 내 검색 → P@3_i = 2/3 기여
  - 24개 쿼리: primary만 top-3 → P@3_i = 1/3 기여
  - 3개 쿼리: recall_at_5=0.5 (secondary top-5 밖) → P@3_i = 1/3 기여
```

v0.8.1 대비 secondary top-3 적중이 5개→3개로 감소. graph_context가 sibling 파일 chunk를 추가 주입하여 cross-encoder 재랭킹 시 secondary 후보와 경쟁.

### MRR@10 불변

Primary qrel은 항상 rank-1 → MRR@10 0.950 유지.

## 결론

TypeDB graph_context 활성화 효과:

- **긍정**: Recall@5 +0.017, nDCG@10 +0.017 — graph expansion이 놓쳤던 secondary를 top-5로 끌어올림
- **중립**: MRR@10 불변 — primary rank는 영향 없음
- **부정**: P@3 -0.022 — graph 결과가 top-3 슬롯을 경쟁하여 secondary 일부 탈락

**TypeDB graph_context는 Recall@5 기준 유효한 소폭 개선**을 보임. P@3 하락은 graph 결과 삽입 순서/수량 튜닝으로 개선 가능성 있음.
